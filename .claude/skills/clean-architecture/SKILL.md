# Clean Architecture — Jurepi 적용

이 스킬은 Jurepi를 **계층화된 의존성 그래프**로 다룬다. 목표는 단순하다: *비즈니스 규칙을 프레임워크로부터 보호*해서 Next.js·Tailwind·AdSense가 바뀌어도 도메인이 흔들리지 않고, 도메인이 결정적이라 TDD가 쉬워지게 한다.

## 왜 이 프로젝트에 클린 아키텍처인가

사다리 **공정성**은 이 제품의 신뢰 그 자체다. 공정성 로직이 React 컴포넌트 안에 얽히면 테스트도, 증명도, 재사용도 불가능하다. 공정성 엔진을 순수 도메인으로 떼어내면 시드 RNG로 10만 회 chi-square를 돌려 *증명*할 수 있다. 같은 원리가 검색·동의·레지스트리에도 적용된다. 그래서 계층을 나눈다 — 미적 취향이 아니라 검증 가능성 때문이다.

## 4계층과 의존성 규칙

**의존성 규칙(불변):** 소스 코드 의존성은 **항상 안쪽으로만** 향한다. 안쪽은 바깥을 모른다.

```
┌─────────────────────────────────────────────────────────┐
│ 4. Frameworks & Drivers  (app/, Tailwind, next-intl 런타임,│
│    AdSense, crypto, localStorage, 브라우저)                │  ← 세부사항
│  ┌───────────────────────────────────────────────────┐  │
│  │ 3. Interface Adapters (React 컴포넌트, 훅, lib/seo,  │  │
│  │    storage 어댑터, i18n 어댑터)                       │  │
│  │  ┌─────────────────────────────────────────────┐  │  │
│  │  │ 2. Use Cases (순수 reducer, 셀렉터, 검색/      │  │  │
│  │  │    필터/동의 게이팅 흐름)                       │  │  │
│  │  │  ┌───────────────────────────────────────┐  │  │  │
│  │  │  │ 1. Domain/Entities (ladder 엔진, 레지스 │  │  │  │
│  │  │  │    트리 타입·불변식, 순수 규칙)          │  │  │  │
│  │  │  └───────────────────────────────────────┘  │  │  │
│  │  └─────────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

| 계층 | 소유 | import 허용 | import 금지 |
|------|------|------------|------------|
| 1 도메인 | 순수 함수·타입·불변식 (`src/lib/ladder.ts`, `src/tools/types.ts`, 검색/동의 순수 판정) | 표준 JS, 같은 계층 | `react`, `next`, `next-intl`, DOM/브라우저 API |
| 2 유스케이스 | 순수 상태머신 reducer, 셀렉터, 흐름 조합 | 도메인 | `react`, `next` (훅 금지 — reducer는 순수) |
| 3 어댑터 | React 컴포넌트·훅, `lib/seo.ts`, storage/i18n 어댑터 | 유스케이스, 도메인, `react` | 다른 어댑터의 내부 구현에 강결합 |
| 4 프레임워크 | `app/`, 설정, 외부 SDK 배선 | 모든 안쪽 + 프레임워크 | (없음 — 가장 바깥) |

> 상세 파일별 매핑과 "이 import는 합법인가?" 판정표는 `references/layer-map.md`를 읽어라.

## 경계 횡단 = 포트 & 의존성 주입

안쪽 계층이 바깥 능력(난수·저장소·시간)이 필요하면, **안쪽이 인터페이스(포트)를 정의**하고 바깥이 구현(어댑터)을 주입한다. 제어 흐름은 바깥→안이지만, 소스 의존성은 안→인터페이스로 유지된다(의존성 역전).

```typescript
// 도메인이 포트를 정의 — 구현은 모른다
type Rng = () => number;                  // [0,1) 균등
export function uniformPermutation(n: number, rng: Rng): number[] { /* 순수 */ }

// 프레임워크가 어댑터를 주입
const cryptoRng: Rng = () => { const b = new Uint32Array(1); crypto.getRandomValues(b); return b[0]/2**32; };
// 테스트는 시드 어댑터를 주입 → 결정적
const seeded: Rng = mulberry32(42);
```

저장소도 같다: 동의/테마 판정은 순수 도메인, `localStorage`는 `useLocalStorage` 어댑터 뒤에 가둔다. 이렇게 하면 도메인 테스트에 브라우저가 필요 없다.

## 미래 백엔드 확장 지점 (YAGNI 준수)

현 SPEC는 백엔드/DB가 없다. **지금 백엔드를 만들지 않는다.** 단, 데이터 접근이 필요해질 때를 위해 *경계만 비워둔다*: 도구가 데이터를 읽어야 하면 도메인이 리포지토리 포트(`findAll/findById/...`)를 정의하고, 어댑터는 처음엔 컴파일타임 레지스트리, 나중엔 API/DB로 바꿔 끼운다. 도메인·UI는 변하지 않는다. **포트를 미리 만들지도 말 것** — 실제 두 번째 데이터 소스가 등장할 때 도입한다.

## 적용 체크리스트

- [ ] 도메인/유스케이스 파일에 `react`/`next`/DOM import가 없는가
- [ ] 비즈니스 규칙이 컴포넌트가 아니라 도메인/유스케이스에 있는가
- [ ] 난수·저장소·시간이 주입되는가(하드 의존 아님)
- [ ] 상태 전이가 순수 reducer로 추출되어 React 없이 테스트 가능한가
- [ ] 새 파일이 올바른 계층에 있고 허용된 것만 import하는가
- [ ] 프레임워크 교체가 도메인을 건드리지 않는가

## 안티패턴 (하지 말 것)

- 컴포넌트 `useEffect` 안에서 공정성/검색 알고리즘을 직접 구현 → 도메인으로 추출.
- 도메인 함수가 `localStorage`/`window`/`crypto`를 직접 호출 → 포트로 주입.
- 라우트 핸들러·페이지에 비즈니스 로직 인라인 → 도메인 호출로 대체.
- "나중에 필요할 것" 추상화 선제 도입 → YAGNI.
